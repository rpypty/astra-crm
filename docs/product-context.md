# Product Context — p2p-crm

## 1. Что делаем

**p2p-crm** — CRM для тимлидов и P2P-трейдеров.

Система нужна для операционного контроля:

- кто из трейдеров с какими реквизитами работает;
- какие обороты фиксируются в течение смены;
- какие входящие ордера пришли из внешней админки;
- какие выплаты трейдер завёл вручную;
- как CSV из внешней админки сходится с ручными данными трейдера;
- где есть расхождения и кто их подтвердил;
- сколько оборота и потенциальной ЗП у трейдера;
- как тимлид в конце периода закрывает сверку по команде.

Это не просто CRM-табличка. Это система учёта смен, реквизитов, CSV-импортов и сверок.

## 2. MVP scope

### Делаем

- Ролевая модель `TEAMLEAD` / `TRADER`.
- Авторизация через логин/пароль.
- Сессии через server-side sessions.
- Панель тимлида.
- Панель трейдера.
- Управление трейдерами.
- Управление реквизитами.
- История назначения реквизита трейдеру.
- Смена трейдера.
- Взятие реквизита в работу.
- Ежедневные данные по реку: карта/ФИО.
- Cumulative-обороты по реку.
- Ручные payout orders и intermediate transfers.
- CSV import для inbound/outbound ордеров.
- Reimport CSV с полной заменой активного набора данных в scope.
- Reconciliation смены.
- Reconciliation периода тимлида для inbound.
- Dashboard/read-модели.
- Audit log.

### Пока не делаем

- Superadmin/admin-панель.
- Микросервисы.
- Очереди/Kafka/RabbitMQ.
- Real-time WebSocket обновления.
- Notification listener APK.
- Browser extension.
- Банковские интеграции.
- Автоматический парсинг пушей.
- Multi-tenant billing.
- Сложную систему прав поверх двух ролей.

## 3. Роли

## TEAMLEAD

Тимлид управляет командой.

Основные страницы:

1. Дашборд.
2. Реквизиты.
3. Сотрудники/трейдеры.
4. Входы/платежи.
5. Выплаты.
6. Периоды.
7. Аудит.

Возможности:

- создать/редактировать/архивировать реквизит;
- назначить реквизит трейдеру;
- посмотреть историю изменения назначения;
- создать/редактировать/деактивировать трейдера;
- задать/сбросить пароль трейдера;
- задать процент ЗП трейдера;
- посмотреть назначенные трейдеру реки;
- импортировать CSV с входящими ордерами за период;
- импортировать CSV с выплатами за период;
- увидеть сверку периода;
- увидеть аналитику по успехам/неуспехам/конверсии;
- посмотреть audit log.

## TRADER

Трейдер работает только со своими назначенными реквизитами и сменой.

Основные страницы:

1. Мои реквизиты.
2. Входы.
3. Выплаты.
4. Аналитика/смена.

Возможности:

- видеть назначенные реквизиты;
- взять реквизит в работу, указав номер карты/ФИО;
- в течение дня добавлять cumulative-оборот по реквизиту;
- импортировать CSV входящих ордеров в конце смены;
- увидеть сверку CSV vs cumulative-обороты;
- если есть расхождение — увидеть красный alert и подтвердить только с обязательным комментарием;
- вручную создавать выплатные ордера;
- наполнять выплату промежуточными переводами;
- импортировать CSV выплат в конце смены;
- увидеть сверку CSV выплат vs intermediate transfers;
- закрыть смену только когда все правила выполнены.

## 4. Главный бизнес-процесс трейдера

```text
1. Трейдер логинится.
2. Видит список назначенных реквизитов.
3. Открывает реквизит.
4. Указывает карту/ФИО.
5. Система автоматически создаёт открытую смену, если её ещё нет.
6. В течение дня трейдер вводит cumulative-обороты по реквизитам.
7. По выплатам трейдер создаёт ручные payout orders.
8. По каждой выплате добавляет intermediate transfers с указанием исходного река и суммы.
9. В конце смены импортирует inbound CSV.
10. Система сверяет successful CSV amount vs latest cumulative turnover by requisites.
11. В конце смены импортирует outbound CSV.
12. Система сверяет payout CSV amount vs sum of intermediate transfers.
13. Если всё совпало — смена закрывается.
14. Если есть расхождение — трейдер должен исправить или принять с обязательным комментарием.
15. Если расхождение принято — смена закрывается как closed_with_discrepancy.
```

---

## 5. Главный бизнес-процесс тимлида

```text
1. Тимлид логинится.
2. Создаёт трейдеров и задаёт им login/password/salary percent.
3. Создаёт реквизиты.
4. Назначает реквизиты трейдерам.
5. В конце периода создаёт/открывает accounting period.
6. Загружает inbound CSV за период.
7. Система сравнивает teamlead period CSV vs trader shift imports.
8. Если есть расхождения — показывает breakdown.
9. Тимлид принимает решение по периоду.
10. Загружает outbound CSV за период, пока без дополнительных триггеров.
```

## 6. Product decisions already resolved

1. Обороты по реку — cumulative.
2. Смену трейдер фиксирует сам через взятие реквизита в работу.
3. Трейдер не может “забыть” открыть смену, потому что она создаётся при заполнении данных река.
4. `workerName` из CSV всегда однозначно соответствует одному трейдеру.
5. Реквизит может переходить между трейдерами внутри дня.
6. `corrected` считается успешным оборотом вместе с `hand_success`.
7. Сверка выплат идёт с intermediate transfers.
8. Нельзя закрыть смену, если payout order заполнен промежуточными переводами не полностью.
9. Комментарий при любом расхождении обязателен.
10. Закрытую смену нельзя переоткрыть.
11. CSV выплат имеет такую же структуру, как CSV входов.
12. `workerAmount`, `workerProfit` пока просто сохраняем.
13. CSV можно перезагружать много раз.
14. Переимпорт CSV должен триггерить пересчёт ордеров с полной заменой активного набора повторяющихся данных.

## 7. UX principle

UX должен быть очевидным и операционно безопасным:

- пользователь всегда понимает, что именно сейчас сверяется;
- успешные/опасные состояния визуально отличаются;
- расхождения нельзя скрыть;
- закрытие смены — не одна “магическая кнопка”, а понятный checklist;
- импорт CSV всегда показывает результат: rows, totals, statuses, mismatches;
- повторный импорт явно показывает, что предыдущий активный набор будет заменён;
- все destructive actions требуют подтверждения.
